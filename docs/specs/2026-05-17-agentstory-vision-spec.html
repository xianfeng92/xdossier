§ 0 · 一句话 把每一次和 AI 协作的过程，自动产出一份你的同事 / 老板 / 未来的你都看得懂的 HTML 报告 —— git for AI thoughts。

§ 1为什么这个项目存在

1.1现状的三种"理解 AI 工作"方式都不够好

方式谁在用痛点
翻 chat log大部分 Claude / Cursor 用户线性、冗长、关键决策被淹在水印里、不可扫读、不可分享
看终端 stdoutCLI 用户实时性强，但事后无法回看；信息密度极高，非作者无法理解
让 AI 给一段 summary所有用户损耗大，看不到失败 / 回退 / 分叉，AI 自己描述自己天然有偏

1.2三个真实场景

这三个场景也是 §3.2 用户故事的原型。

  1. A 用 Claude Code 改了一个生产 bug，老板要他写复盘 → 现在的做法是手动回忆 + 翻 git log + 拼凑文字。
  2. B 用 Cursor 做了一周研究笔记和实验，想分享给团队 → 没有合适的"过程档案"格式，只能贴 chat 截图或重写一遍。
  3. C 反复迭代了 5 版简历（DecisionF 场景），想看哪一次改动是关键转折 → 现在完全没有工具。

1.3设计原点

Thariq Shihipar 的 Unreasonable Effectiveness of HTML 提出：HTML 比 Markdown 适合 AI 输出，因为 HTML 能承载空间结构而非把信息拍扁成线性文本。但他主要谈"AI 最终产物"。本项目把这个思路 扩展到 AI 的工作过程本身 —— 过程也是空间结构（timeline、swimlane、DAG、diff），不该塞进 chat log。

nexu-io/html-anything 验证了"Markdown → 设计级 HTML"的产品形态可行，并贡献了：(a) 多 agent CLI 检测与统一调用层，(b) SKILL.md 协议作为可复用的输出模板单元。本项目借用其底层架构 + SKILL 协议，但目标是不同的人群和不同的产出形态。

§ 2这个项目不做什么

跟"做什么"一样重要 —— 用 non-goals 反向定义边界。

❌ 不做因为
✗开发者 observability dashboard（Langfuse / LangSmith / Phoenix Arize 形态）那是为 agent 开发者调试用的；本项目为使用 agent 把工作做完的人服务
✗又一个 chat history viewer现存的太多，只是把 JSONL 漂亮地铺平没有意义
✗新的 agent runtime / 新的 CLI agent不和 Claude Code / Cursor / Codex 竞争，只消费它们的 trace
✗"如何让 AI 干得更好"那是 prompt engineering / agent tuning 领域；本项目只解决"事后如何让人理解 AI 干了什么"
✗实时监控 / 告警实时性不是核心价值，事后可读、可分享、可回溯才是
✗多人协作 / 评论 / 工作流第一版纯个人本地；多人协作放在 v2+
✗SaaS / cloud-only本地优先，所有数据默认不出本机（同 html-anything 的哲学）

§ 3目标用户的精确定义

3.1谁是"使用者"

用 agentic CLI（Claude Code、Cursor Agent、Codex、Gemini CLI 等）把工作做完，事后自己想复盘、或需要给非 AI-原生的人解释 AI 工作过程的人。

精确刻画：

  • ✓ 会：读 git diff、看 tool name、理解"AI 调用了 Read / Edit / Bash"
  • ✓ 会：用 Markdown 写文档
  • ✗ 不会 / 不想：读千行 JSONL trace、配置 OTel collector、写 Python 解析器

注意：这不包括 chat.claude.ai 网页用户（他们不接触 trace 这个概念），但预留它的扩展。

3.2用户故事（MVP 必须显式覆盖）

US-1 · 复盘故事
A 修了一个生产 bug 用了 2 小时 Claude Code session
事后 A 跑一个命令 → 浏览器自动打开一份 HTML 报告 → 顶部时间轴清晰显示"先调查 → 试错两次 → 找到根因 → 修复 → 验证"，每个阶段可展开看具体 tool call 和 diff → A 把这个文件直接发给老板。
US-2 · 学习故事
C 想知道自己 5 次简历迭代里哪一次改动最关键
跑一个命令把 5 个 session 合并成一份"演化报告" → 看到第 3 次迭代时 AI 把"项目经验"重构成了 STAR 格式，这是评分跃升的转折。
US-3 · 分享故事
B 想把一周的 NLU 研究分享到博客 / Twitter
一键 export 成可托管的单文件 HTML（或 PNG），不用重新组织一遍内容。

3.3反用户故事（明确不服务）

ANTI · 1
我是 agent 开发者，想知道我的 prompt 在哪一步偏离了。
→ 用 Langfuse。
ANTI · 2
我想要一个 SaaS 协作平台让团队评论 AI 输出。
→ v2+ 再说。
ANTI · 3
我从来不用 CLI agent，只用 ChatGPT 网页版。
→ 不是本期目标用户。

§ 4核心架构（三层）

┌─────────────────────────────────────────────────────┐
│  L1 Capture（适配器）                               │
│  Claude Code | Cursor | Codex | Gemini | Aider ...  │
│           ↓ 每个适配器输出统一格式                  │
├─────────────────────────────────────────────────────┤
│  L2 Trace IR（开放数据模型，本项目的"协议"）        │
│  events[] · decisions[] · artifacts[] · narrative   │
│           ↓                                         │
├─────────────────────────────────────────────────────┤
│  L3 Render（profile + skill + 模板/LLM 混合）       │
│  → 单文件 HTML（可分享、可托管）                    │
└─────────────────────────────────────────────────────┘
关键设计判断：L2 Trace IR 是项目的"宪法"。Capture 适配器和 Render skill 都围绕它演化。把 IR 设计好，是这个项目长期可维护的根。

§ 5数据源与捕获策略

策略：被动优先，主动可选。

档数据来源信息密度用户操作
L1 被动捕获 Claude Code ~/.claude/projects/<encoded>/*.jsonl、Cursor 本地 chat 存档、Codex session log 高 0 — 自动扫描
L2 工作目录 diff git log --since=<session-start> + 文件 mtime + 未追踪文件 中（产物 ground truth） 0 — 自动关联
L3 主动埋点 settings.json hooks（PostToolUse / Stop）/ MCP server / wrapper CLI 极高（显式决策点） 一次配置

关键策略：MVP 只做 L1 + L2，且只支持 Claude Code 一个适配器。验证完价值再扩展。

L1 + L2 的关联机制：用 session 起止时间 + 当前工作目录的 git HEAD 做时间窗口对齐，把 chat trace 的每个 tool call 链接回 git diff 的具体 hunk，建立 artifact ↔ reasoning step 的双向 provenance。这是本项目相对所有现存工具的核心差异化能力。

§ 6Trace IR 数据模型（草稿）

type TraceIR = {
  session: {
    id: string;
    agent: "claude-code" | "cursor" | "codex" | ...;
    started_at: ISO8601;
    ended_at: ISO8601;
    cwd: string;
    git_head_before: string;   // commit hash
    git_head_after: string;
    user_initial_request: string;
  };

  events: Event[];               // 时间序列，原始流
  decisions: Decision[];         // 抽取后的决策点（LLM 二次处理）
  artifacts: Artifact[];         // 产出物（文件 / 代码 / HTML 等）
  narrative: Narrative;          // LLM 生成的故事化摘要
  cost: { tokens, dollars? };
};

type Event =
  | { type: "user_message"; text: string; ts: ISO8601 }
  | { type: "assistant_message"; text: string; ts: ISO8601 }
  | { type: "tool_call"; tool: string; input: unknown; ts: ISO8601; call_id: string }
  | { type: "tool_result"; call_id: string; output: unknown; ts: ISO8601 }
  | { type: "error"; message: string; ts: ISO8601 };

type Decision = {
  id: string;
  ts: ISO8601;
  summary: string;               // "选择用 zustand 而非 redux"
  rationale: string;             // 抽取自 assistant prose
  alternatives_considered: string[];
  source_event_ids: string[];    // 反向定位
  artifacts_affected: string[];  // 影响了哪些产出
};

type Artifact = {
  id: string;
  path: string;                       // 相对工作目录
  kind: "file" | "html" | "code" | "doc";
  change_type: "created" | "modified" | "deleted";
  versions: { ts: ISO8601; content_hash: string; source_event_id: string }[];
  provenance: string[];               // 哪些 decision/event 产生了它
};

type Narrative = {
  summary: string;                // 100 字总览
  acts: { title: string; ts_range: [ISO8601, ISO8601]; description: string }[];
  turning_points: { ts: ISO8601; why_important: string }[];
};

关于决策抽取：是 LLM 后处理 pass（cheap model 一次调用）。MVP 阶段允许"准确率不高"，因为有原始 events[] 兜底。

§ 7渲染层：Profile + Skill + Memory 三层个人化

7.1Layer 1 — Profile（开箱即用，5-6 个内置）

Profile适用场景默认 section决策抽取重点
coder写代码 / 改 bugtimeline · diff · decisions · cost选型、回退、bug 根因
writer写文档 / 文章timeline · artifact diff · narrative语气、结构调整
researcher论文阅读 / 调研citations · concept map · narrative论点演化
designer设计 / UI 迭代artifact gallery · visual diff风格选择
reviewer代码 reviewdiff + severity · jump linksreview 决策
learner学习 / 教程step-by-step · 知识点折叠困惑与厘清的瞬间

7.2Layer 2 — Skill（SKILL.md 模式，社区可扩展）

借鉴 html-anything 的 SKILL.md：每个"trace 视图"是一个目录。

agentstory/skills/
├── trace-coding-session/        # 编码 session 默认皮肤
│   ├── SKILL.md                  # 渲染指令（prompt + 模板路径）
│   ├── template.html             # 静态骨架
│   └── example.html              # 样例输出（同 html-anything 惯例）
├── trace-resume-iteration/       # 简历迭代专属（DecisionF）
├── trace-paper-reading/          # 读论文专属
├── trace-bug-postmortem/         # 复盘专属
└── ...

7.3Layer 3 — Personal Memory（真正个人化的一层）

~/.agentstory/profile.md —— 人类可读的偏好文件：

# My report preferences

- 总是显示：tool timeline、git diff
- 默认折叠：raw reasoning（除非我点开）
- 我特别关心：当 AI 推翻自己之前判断的时刻、当 AI 询问澄清的时刻
- 我不关心：token 成本、模型选择细节
- 渲染语言：中文，但代码 / 文件路径 / 命令保持英文

关键设计选择：profile 是 markdown，不是 JSON。

  • 用户可以让 AI 帮自己改 profile："帮我把研究 session 的报告改得更像论文 review"
  • 反馈闭环：每份报告底部一个反馈条 → 一句话写回 profile → 下一份自动更准
  • profile 本身也进入 AI 工作流

7.4渲染实现：模板 + LLM 混合

  • 结构化骨架（timeline / diff / artifact gallery / decision table）→ 静态模板（jinja-like），快、便宜、确定
  • 叙事性段落（narrative.summary / acts / turning_points）→ LLM 一次调用，复用 Profile + Memory 作为 system prompt

§ 8嵌入 AI 工作流的四档

按摩擦力递增。

档用户操作适合谁
1 · 零配置 post-hocagentstory open → 自动找最近 Claude session → 浏览器打开默认入口，80% 用户
2 · 三行 hook 集成~/.claude/settings.json 加一个 Stop hook → session 结束自动出报告想要自动化的人
3 · CLI wrapperagentstory claude -p "..." → 实时画 timeline喜欢看 AI 干活
4 · MCP serveragent 主动调 record_decision / mark_turning_point严肃工作（合规、研究、面试）
关键原则：默认是档 1。每多一档解锁能力，但绝不强制升档。哲学等同 OpenTelemetry 的 "auto-instrument first, manual when needed"。

§ 9MVP 范围（P0）

目标：6-8 周做一个能让 §3.2 US-1（复盘故事）跑通的最小版本。

包含（P0）

  • 仅支持 Claude Code 一个 Capture 适配器
  • Trace IR 完整定义 + 第一版实现
  • git diff 与 trace 时间窗口的关联机制
  • 决策抽取的第一版（LLM 后处理）
  • 1 个内置 profile（coder）+ 1 个 skill（trace-coding-session）
  • 嵌入档 1（post-hoc）+ 嵌入档 2（hook）
  • 单文件 HTML，可在浏览器打开 / 拷贝分享
  • Personal memory 极简版（一份 profile.md，无反馈条 UI）

推迟到 P1+

  • Cursor / Codex / Gemini 适配器
  • MCP server（嵌入档 4）
  • CLI wrapper（嵌入档 3）
  • 多 session 合并 / 演化报告（US-2 暂不覆盖）
  • Export 到 PNG / WeChat / X（US-3 暂不覆盖；可复用 html-anything 的 export 管线，但 P1 才做）
  • 反馈条 UI 闭环

§ 10与 html-anything 的关系

独立项目，但共享底层。

  • 可复用：html-anything 的 multi-agent 检测层（src/lib/agents/detect.ts）、SKILL.md 协议、SSE 流式渲染、sandboxed iframe 预览模式、export 管线（juice / modern-screenshot / remotion）。
  • 不复用：UI 形态完全不同。html-anything 是"编辑器 + 实时预览"，本项目是"事后报告浏览器"。
  • 战略关系：html-anything 产出"对外发布的设计级 artifact"；agentstory 产出"内部协作的 session report"。两者互补不竞争。

实施建议：MVP 阶段可以单独立项（不进 html-anything 仓库），但显式声明"build on top of html-anything（同 html-anything build on top of open-design）"，并贡献回任何通用层改进。

§ 11开放问题

必须在 MVP spec 前回答。

Q1技术栈：Next.js（同 html-anything）/ pure TS CLI + 静态 HTML 输出 / Rust + WASM ？
pure TS CLI，产出物是单文件 HTML（不要 server）。但 Render 阶段可能想要个本地预览 server。
Q2决策抽取的 LLM 是哪个？默认走用户当前 agent，还是固定一个 cheap 模型？
默认复用用户 logged-in 的 agent（同 html-anything 哲学，0 API key）。允许覆盖。
Q3跨 session 的"演化报告"在 P0 留接口还是直接 punt？
留接口（IR 设计支持 multi-session 合并），实现 punt 到 P1。
Q4HTML 报告是单文件 inlined CSS（同 html-anything）还是带 assets 的目录？
单文件。可分享是核心价值。
Q5Cursor session log 的格式是否稳定？需要 reverse engineer 还是它有官方协议？
待调研。MVP 不涉及。
Q6隐私默认值：trace 中包含的 Bash 命令 / 文件内容里有 secrets，默认 redact 还是默认显示？
默认显示（用户本地工具），提供 redact-patterns 配置文件支持显式打码（典型 patterns: .env* 文件内容、API_KEY 等环境变量、OAuth 类 token）。
Q7报告底部的"反馈条 UI 写回 profile.md"是 P0 还是 P1？
P1。P0 让 profile.md 纯手写。

§ 13成功标准（项目级，不是 MVP 级）

按可验证程度排序。

最弱能跑通 §3.2 三个用户故事至少一个。
中等自己每天的 Claude Code session 出报告，且真的去点开看（而不是装好就忘）。
较强把一份报告发给一个非 AI-原生的朋友 / 同事 / 老板，对方能在 3 分钟内说清楚"AI 干了啥"。
强有第二个用户写了一个自己的 trace skill 并 PR 回来。
最强被一个其他 agent CLI（Cursor / Codex / Gemini）官方推荐为"事后报告工具"。

§ 14风险与开放性问题（vision 层）

风险缓解
Claude Code 的 JSONL 格式变适配器层隔离；尽快推进 trace IR 作为开放协议
决策抽取不够准events[] 永远作为兜底；用户可在报告上 reroll 一段
被当成"Langfuse 的廉价版"文档反复强调 §3.3 反用户故事；UI 风格刻意走"故事 / 杂志"而非"dashboard"
个性化做得太复杂用户配不动默认 profile 必须能直接用；profile.md 是 markdown，能让 AI 帮改
隐私顾虑本地优先；redact 默认开启；显式列出 ~/.agentstory/redact-patterns.json
